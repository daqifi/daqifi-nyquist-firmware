# Flash Wait-State (PFMWS) tuning at 252 MHz — investigation (#586)

**Date:** 2026-07 · **Refs:** #487 (SYSCLK 200→252 MHz), #586
**Scope:** determine the correct `PRECONbits.PFMWS` value for the 252 MHz
operating point with Flash ECC enabled (`CFGCONbits.ECCCON = 3`), and decide
whether to lower it from the conservative bring-up value of 5.

**Verdict: recommend `PFMWS = 4` (doc-only for now).** The datasheet + errata
minimum with ECC is **4**, not 3 — 3 would be *too few* and risk flash-fetch
faults. Lowering is the risky direction; commit the code change only after an
at-speed bench soak (see "Validation required" below). No firmware value is
changed by this document.

---

  #define DAQIFI_PFMWS  5u   /* flash wait states (bring-up conservative) */
#else            /* 200 MHz legacy */

```c

PRECONbits.PFMWS  = DAQIFI_PFMWS;
CFGCONbits.ECCCON = 3;          /* Flash ECC enabled */
```

- **V** (firmware): 200 MHz point runs `PFMWS = 3` with `ECCCON = 3`, and boots
  + soaks clean (shipped since #487 era). This is our trusted anchor.
- **V** (firmware): 252 MHz point runs `PFMWS = 5`, chosen "conservatively high
  for the first bring-up so the CPU can never fault on a flash read" (comment,
  `initialization.c`). The reset-default `PFMWS` is the maximum (7), so this
  only affects steady-state performance, never the boot window.

## 2. The rule (X)

**Program Flash Memory wait states** insert `PFMWS` extra SYSCLK periods per
flash fetch, so the flash is given `(PFMWS + 1)` SYSCLK periods to return data.
Effective flash access frequency is therefore:

```
f_flash_eff = SYSCLK / (PFMWS + 1)
```

`f_flash_eff` must stay at or below the flash array's rated access frequency.
The PIC32MZ EF flash array tops out around ~66–75 MHz per access (i.e. one
0-wait-state fetch is valid only up to that SYSCLK) — this is the cadence the
DS60001320 "Program Flash Memory Wait States" electrical table encodes.

**Errata DS80000663 #38 (X, System Bus):** *"Flash wait states at SYSCLK
>184 MHz with ECC need 3 wait states (not 2). <2% CPU impact with cache."*
Flash **ECC adds one cycle of read latency** beyond the raw array timing, so an
ECC-enabled part needs **one more wait state** than the bare-flash table implies
at the same SYSCLK. This is exactly why the 200 MHz point runs 3 and not 2.

## 3. Deriving the 252 MHz value (I, three concordant methods)

### Method A — errata step from the trusted 200 MHz anchor
At 200 MHz the raw-flash requirement is `PFMWS = 2` (eff 66.7 MHz), and errata
#38 bumps it to **3** for ECC. Moving 200 → 252 MHz is one wait-state cadence
step (~+60 MHz), so the raw requirement rises 2 → 3, and the ECC bump makes the
enabled value **3 → 4**.

### Method B — effective-access ceiling (`SYSCLK/(PFMWS+1)`)

| SYSCLK | PFMWS | flash periods | eff access | vs ~66–75 MHz array + ECC latency |
|-------:|------:|--------------:|-----------:|-----------------------------------|
| 252 MHz | 2 | 3 | 84.0 MHz | over the array ceiling — **unsafe** |
| 252 MHz | 3 | 4 | 63.0 MHz | OK for *bare* flash, **too few with ECC** (no room for the +1 ECC cycle) |
| 252 MHz | **4** | 5 | 50.4 MHz | **safe with ECC** — datasheet+errata minimum |
| 252 MHz | 5 (current) | 6 | 42.0 MHz | safe, one extra wait state of headroom |

The 200 MHz anchor calibrates this: at 200 MHz, `PFMWS = 2` gives 66.7 MHz
(right at the bare-array edge) and errata #38 requires 3 for the ECC cycle. The
same +1-for-ECC margin at 252 MHz lands on **4**.

### Method C — cadence extrapolation (the #586 ticket's own note)
"A ~60 MHz-per-wait-state cadence would put 252 MHz at 4." Concordant with A
and B.

**All three land on `PFMWS = 4` with ECC.** The current `5` carries exactly one
wait state of margin above the minimum.

> ⚠️ **`PFMWS = 3` (the value guessed by the tune-down task) is NOT safe with
> ECC enabled.** 3 gives a 63 MHz effective access that leaves no room for the
> ECC read-latency cycle that errata #38 exists to account for. Do not set 3.

## 4. Expected benefit (I)

Going 5 → 4 removes **one** wait state from every flash fetch that is *not*
absorbed by prefetch (`PREFEN = 3`) or the 16 KB L1 I-cache. On this firmware
the hot loops (streaming deferred task, encoders, ISRs) run almost entirely from
I-cache, and linear code is prefetched, so the *measurable* delta is expected to
be **well under 1%** — errata #38 itself quotes "<2% CPU impact with cache" for
a whole wait-state at this class of frequency, and that is the delta between the
*maximum* (7) and the low end, not between 5 and 4. The benefit of 5 → 4 is real
but small, and free. It is **not** a throughput lever for the streaming caps.

## 5. Risk & recommendation

- **Risk direction:** lowering is the dangerous direction. Too few wait states →
  the CPU latches flash data before the array (plus ECC) is valid → corrupted
  instruction/constant fetch → exception or silent misbehavior that *looks like
  a brick*; recovery is a reflash. The reset default (7) protects the boot
  window, so a bad value manifests only after `initialization.c` runs.
- **Confidence in 4:** high on the *relative* argument (one step above the
  trusted 200 MHz/ECC=3 anchor, +1 for ECC — Methods A/B/C agree). The exact
  DS60001320H electrical-table breakpoints were reconstructed from the anchors
  and the ~66–75 MHz array cadence rather than read from the PDF in this
  session; a final cross-check against DS60001320H's actual "Program Flash
  Memory Wait States" table is the one open item before flipping the code.

**Recommendation:** lower `DAQIFI_PFMWS` (252 branch) from `5` to `4` **after**
(a) confirming the value against the DS60001320H PFMWS electrical table, and
(b) an at-speed bench soak. Until then, keep 5 — the extra wait state is
harmless headroom and the device is not flash-fetch-bound.

## 6. Validation required before the code change lands

Flip `DAQIFI_PFMWS` 5 → 4 on a branch and, on real hardware at 252 MHz:

1. **Boot + enumerate** clean across ≥10 power cycles (a marginal wait state
   often shows as intermittent boot hangs / TLBL exceptions, not a hard fail).
2. **Long streaming soak** (hours) at a high cap, watching for any new
   exceptions or `gCrash*` captures — flash-fetch corruption on a rarely-taken
   code path is the failure mode to rule out.
3. **A/B throughput** at 4 vs 5 (expect it in the noise; if a config shows a
   real delta, note it — but do not treat 5 as a throughput regression).

Keep the reflash-recovery path in mind throughout: a too-few-wait-state build
can present as an unresponsive board.
