# Async / Non-Blocking SD Write Path — Feasibility Study

**Ticket:** #224 — refactor the SD write path from blocking to async/non-blocking
**Status:** Feasibility assessment only (this is a large refactor — scope it, don't implement)
**Firmware baseline:** v3.6.3, PIC32MZ2048EFM144 / FreeRTOS / XC32 v4.60, NQ1/NQ2/NQ3
**Date:** 2026-07
**Author:** engineering (ticket #224 triage)

> **Bottom line up front.** A fully async write path is *feasible* and the
> hardware/driver machinery to do it **already exists** (the Harmony SDSPI
> driver is an interrupt/DMA-driven FSM; `DRV_SDSPI_AsyncWrite` +
> `DRV_SDSPI_CommandStatusGet` are already compiled in). The blocking is **not**
> in the SPI transfer — it is a busy-wait glue loop in the FatFS `diskio.c`
> layer that spins the driver FSM to completion instead of yielding. That
> reframes the whole ticket: the win available is **CPU reclaim during the
> ~1–2 ms per-sector SPI transfer**, not a faster SPI bus. The recommendation
> is a **phased** approach: first quantify the actual stall (Phase 0), then land
> the low-risk cooperative-yield change (Phase 1) before considering the
> high-risk FatFS-bypass rearchitecture (Phase 3), which is **not recommended**
> without a hard, measured throughput requirement that Phase 1 cannot meet.

---

## 1. Current blocking flow (verified)

The streaming-to-SD write path, top to bottom:

```
streaming_Task (pri 6)  — encodes sample, writes bytes into SD circular buffer
        │  (StreamingBufferPool SD circular partition, 512 B .. 32 KB)
        ▼
app_SDCardTask (pri 5)  — SD_CARD_MANAGER_PROCESS_STATE_WRITE_TO_FILE
        │  sd_card_manager.c:1085  (chunk loop, up to MAX_CHUNKS_PER_CYCLE)
        │  extracts sector-aligned bytes → writeBuffer (coherent DMA buffer)
        ▼
SDCardWrite()            — sd_card_manager.c:206
        │  SYS_FS_FileWrite(fileHandle, writeBuffer+offset, writeBufferLength)
        ▼
FatFS f_write            — updates FIL fptr/cluster chain, calls disk_write per sector run
        ▼
disk_write()             — diskio.c:228
        │  SYS_FS_MEDIA_MANAGER_SectorWrite(...)   ← submits, returns immediately
        │  disk_checkCommandStatus(pdrv)           ← THE BLOCK
        ▼
disk_checkCommandStatus  — diskio.c:54
        while (commandStatus == SYS_FS_MEDIA_COMMAND_IN_PROGRESS) {
            SYS_FS_MEDIA_MANAGER_TransferTask(pdrv);   // spins the SDSPI FSM
        }
```

### 1.1 Where the block actually lives — and where it does NOT

**Verified (V):**

- `disk_checkCommandStatus` (`diskio.c:67-70`) is a **hard busy-wait**: it loops
  calling `SYS_FS_MEDIA_MANAGER_TransferTask(pdrv)` until the media reports
  `COMMAND_COMPLETED`. No `vTaskDelay`, no yield — the SD task holds the CPU
  (at priority 5) for the entire sector-write duration.
- The underlying SPI transfer is **already interrupt/DMA-driven**, *not*
  bit-banged synchronously: the SDSPI driver registers an event handler with
  the SPI driver — `DRV_SPI_TransferEventHandlerSet(dObj->spiDrvHandle,
  DRV_SDSPI_SPIDriverEventHandler, ...)` at `drv_sdspi.c:1705`. The SPI DMA
  completion fires the handler, which advances the SDSPI FSM.
- The SDSPI FSM (`DRV_SDSPI_Tasks`, `drv_sdspi.c:2583`) is a cooperative state
  machine. **It is driven exclusively from the SD task** — `app_freertos.c:330`
  and `:420` call `DRV_SDSPI_Tasks(sysObj.drvSDSPI0)`; the autogenerated
  `tasks.c:74` call is commented out. So today nobody advances the SD media FSM
  except the SD task (either through its main loop, or synchronously through the
  `TransferTask` calls inside the busy-wait).
- Async APIs are present and compiled: `DRV_SDSPI_AsyncWrite` (`drv_sdspi.h:808`)
  and `DRV_SDSPI_CommandStatusGet` (`drv_sdspi.h:1183`).

**Inference (I):** because the transfer itself is DMA/IRQ-driven, the busy-wait
loop is *mostly spinning* — each `TransferTask` call polls FSM state and, when a
DMA transfer is in flight, has nothing to do but return. The CPU the SD task
burns in that `while` loop is largely **reclaimable** if the task yielded and let
the DMA-completion IRQ do the advancing. Confirming the exact reclaimable
fraction is Phase 0 below (the SDSPI FSM also does per-byte
`0xFF`-token busy-status polling of the card between blocks — `drv_sdspi.c`
`DRV_SDSPI_SPIRead(dObj, dObj->pCmdResp, 1)` reads — which *is* real SPI work,
not idle spin; the reclaimable fraction is the difference).

### 1.2 Why it matters (the #224 symptom)

At high streaming rates the SD circular buffer fills during the multi-sector
write stall. Because the SD task is busy-waiting, it cannot re-enter the chunk
loop to drain the circular buffer, and it cannot let the pri-6 encoder run to
push more bytes. The ticket cites 12–13 bytes lost at 5000 Hz (3/10 runs) after
the 20 MHz SPI change (#78) — marginal but real, and exactly the signature of a
producer briefly outrunning a stalled consumer.

Note the current mitigations already in the state machine (all verified in
`sd_card_manager.c`):

- **Sector-aligned extraction** (WRITE_TO_FILE) so FatFS takes the fast
  `disk_write` path (no per-sector memcpy) — the ~55 % throughput win recorded
  in CLAUDE.md.
- **All-or-nothing / partial-write handling**: the chunk loop already tolerates
  `SYS_FS_FileWrite` returning fewer bytes than requested (`writeLen <
  writeBufferLength` → advance `sdCardWriteBufferOffset`, retry).
- **Chunk batching** up to `SD_CARD_MANAGER_MAX_CHUNKS_PER_CYCLE` per task cycle.

Any async design must preserve **all three**, plus the two drain paths that must
*not* sector-align (rotation and UNMOUNT) so sub-sector tails still flush.

## 2. Async design sketch

Three candidate architectures, in increasing order of invasiveness. The
critical constraint that shapes all of them: **the SD media FSM is only advanced
by the SD task** (§1.1). So "issue async and go do other work" only works if the
SD task's *other work* keeps calling `DRV_SDSPI_Tasks` / `TransferTask`.

### Option A — Bypass FatFS for the streaming data path (NOT recommended)

Use `DRV_SDSPI_AsyncWrite` directly for the bulk sample data, keep FatFS only
for metadata (open/close/directory/rotation). The SD manager would own the FAT
cluster walk for the active file: pre-allocate a cluster chain, translate the
file byte offset to an LBA sector itself, and stream sectors straight to the
driver, polling `DRV_SDSPI_CommandStatusGet`.

- **Upside:** maximum throughput; the FatFS `f_write` bookkeeping (fptr, window
  buffer, dirty-flag) is entirely off the hot path; genuine overlap of encode
  and write.
- **Downside:** we now own FAT32 cluster allocation, the FSINFO free-count, the
  directory-entry size update on close, and crash consistency. This is a
  filesystem re-implementation for the write path. It also has to stay correct
  across the existing **file-splitting / rotation** feature and the
  MAXSize logic. High defect surface; a bug corrupts customer data files.
- **Verdict:** disproportionate risk for a 12–13-byte-loss ticket. Reject unless
  a future requirement demands throughput Phase 1/2 provably cannot deliver.

### Option B — Cooperative-yield the media wait (recommended first real change)

Keep FatFS. Restructure the `disk_checkCommandStatus` busy-wait so that, while a
transfer is genuinely in-flight (DMA pending, nothing for the FSM to do), the SD
task **yields** rather than spins. The subtlety (§1.1): the SDSPI FSM must keep
being advanced, so a naive `vTaskDelay(1)` inside `disk_checkCommandStatus` that
*stops* calling `TransferTask` would deadlock — nothing would drive the DMA
completion follow-up. Two viable shapes:

- **B1 — bounded yield inside the wait:** in the `while` loop, call
  `TransferTask` (advance FSM), then if still `IN_PROGRESS` and the SPI layer is
  waiting on DMA, `taskYIELD()` / `vTaskDelay(0)` so equal/higher-priority tasks
  (the pri-6 encoder is *higher* than the pri-5 SD task, so it already preempts;
  the benefit here is letting the encoder run *sooner* and letting other pri-5
  work proceed). Net effect is modest because the encoder already preempts SD.
- **B2 — hoist the wait into the SD task state machine (preferred):** this is
  the honest version of "async." Instead of `SYS_FS_FileWrite` blocking, the SD
  manager issues the write, then **returns to its own task loop** and keeps
  calling `DRV_SDSPI_Tasks` + draining the circular buffer into the *next*
  writeBuffer, polling write completion via a new WRITE_IN_FLIGHT sub-state.
  This requires FatFS to expose a non-blocking write submit — which it does not
  natively; `f_write` is synchronous by contract. So B2 in practice means either
  (a) a small patch to the diskio layer to expose submit/poll to the manager, or
  (b) drifting toward Option A for the data path. B2 is the design that actually
  removes the stall; B1 only softens it.

- **Verdict:** **B1 is the low-risk starting point** (localized to `diskio.c`,
  no filesystem semantics change, tiny RAM cost). **B2 is the target** if Phase 0
  shows B1's reclaim is insufficient — but B2's need for a non-blocking FatFS
  submit is the same architectural boundary Option A crosses, just less of it.

### Option C — Double-buffer (ping-pong) writeBuffer

Orthogonal to A/B and combinable with B2. Today there is one `writeBuffer`
(coherent DMA buffer). With two, the SD task fills buffer *N+1* from the
circular buffer while buffer *N* is being DMA'd out, then swaps. This is the
classic producer/consumer overlap and is the *structural* half of B2.

- **Upside:** overlaps the circular-buffer→writeBuffer memcpy and the extraction
  logic with the in-flight DMA; smooths burst absorption.
- **Downside — RAM (see §4):** a second writeBuffer competes for the 124 KB
  coherent pool, which is already auto-balanced across SD/USB/WiFi DMA. Doable
  by halving the SD DMA share, but that trades burst headroom for overlap; needs
  measurement. No *new* static BSS if it comes from the existing coherent pool
  partition — important given the tight static headroom (§4).

## 3. Benefit estimate

**Honest framing (I, to be confirmed by Phase 0):** the ceiling on this work is
"the CPU the SD task currently burns busy-waiting on DMA." That is bounded by
the *idle-spin* fraction of the ~1–2 ms/sector write, not the whole write.

- If the busy-wait is mostly real SPI work (per-byte card-busy token polling +
  block shifting at 21 MHz), the reclaimable CPU is small and the throughput
  gain is marginal — the win is **latency/jitter smoothing and eliminating the
  12–13-byte tail-loss**, not a higher KB/s ceiling.
- If a meaningful fraction is DMA-wait idle spin, Option B2 + C could let the
  encoder and circular-buffer drain proceed during that window, raising the
  sustainable SD streaming rate. The Session-24 SD soak ceilings
  (CLAUDE.md: e.g. 1×T1 PB 10 kHz, 16ch CSV 1.5 kHz) are the numbers to beat;
  any refactor must be re-soaked against them.

**Do not promise a throughput number before Phase 0.** The #376/#361 WiFi saga
is the cautionary precedent: "it blocks synchronously so removing the block must
speed it up" was repeatedly falsified because the bottleneck was elsewhere. The
SD path deserves the same measured-first discipline. The concrete, defensible
benefit today is **removing the 5 kHz tail-loss** (the actual #224 symptom),
which even Option B1 likely achieves.

## 4. Risk & complexity

| Axis | Option A (bypass) | Option B1 (yield) | Option B2 (SM async) | Option C (double-buf) |
|---|---|---|---|---|
| Files touched | diskio + new FAT logic in sd_card_manager | diskio.c only | diskio + sd_card_manager state machine | sd_card_manager + coherent pool |
| Filesystem semantics | **rewritten** (clusters/FSINFO/dirent) | unchanged | unchanged if submit/poll patched cleanly | unchanged |
| Data-loss / corruption risk | **high** (own FAT correctness) | low | medium (in-flight buffer ownership) | low–medium |
| Concurrency risk | medium | low | medium (WRITE_IN_FLIGHT vs rotation/unmount drains) | low |
| RAM cost | ~0 new | ~0 | ~0 new (state only) | 2nd DMA buf from coherent pool |
| MCC-regen durability | poor (diskio is generated) | **poor — diskio.c is Harmony-generated** | poor | good (manager is ours) |
| Preserves sector-align / all-or-nothing | must re-implement | yes | yes | yes |
| Effort | weeks | days | 1–2 weeks | days |

**Cross-cutting risks:**

- **`diskio.c` is Harmony/MCC-generated.** Per CLAUDE.md, MCC regeneration is
  off the table going forward ("No more MCC" policy), which *reduces* the
  clobber risk — but any diskio edit still lives in generated territory and must
  be documented like the existing generated-file patches. Prefer pushing logic
  into `sd_card_manager.c` (ours) over `diskio.c` where possible (mirrors the
  diskio-metrics refactor lesson in project memory).
- **All-or-nothing write policy.** Project memory: partial writes cause a convoy
  effect; the current design is all-or-nothing on the writeBuffer. An async
  in-flight buffer must not resurrect partial-write fragmentation — completion
  accounting has to be exact (the existing `writeLen >= writeBufferLength` vs
  partial branches are the template).
- **Rotation & UNMOUNT drain paths.** These deliberately do **not** sector-align
  (must flush sub-sector tails) and currently assume writes complete
  synchronously before `f_close`. Any async path must fully **quiesce the
  in-flight write** before rotate/close/unmount, or risk losing the final
  partial sector — the exact data-loss the feature guards against.
- **FPU / task rules.** `app_SDCardTask` is a **pure-integer task** (no
  `portTASK_USES_FLOATING_POINT`). New code in the SD path must not read/cast
  `double`/`float` fields (CLAUDE.md concurrency rules) — trivially satisfied
  here but worth stating for reviewers.
- **32-bit atomicity.** New shared flags between the WRITE_IN_FLIGHT sub-state
  and completion (e.g. a "write pending" word) are fine as plain 32-bit
  reads/writes; any RMW counter needs a critical section (PIC32MZ rules).
- **NQ1/NQ2/NQ3 consistency.** The SD path is board-variant-independent (same
  SDSPI driver, same manager), so no per-variant struct fan-out — but the change
  must be validated on the streaming configs each variant actually uses.

## 5. Phased recommendation

**Phase 0 — Measure the stall (prerequisite, no firmware change beyond
instrumentation).** Before touching the architecture, quantify what fraction of
the sector-write wall time is reclaimable DMA-idle spin vs real SPI work. Tools:
Saleae on SPI4 CS/CLK during a sustained SD stream (bus-idle % during a write,
the way #376 measured WiFi SPI), plus a DIO probe toggled around
`disk_checkCommandStatus`. Deliverable: "X µs of the Y µs/sector write is
reclaimable." This decides whether Phase 2/3 is even worth it. **~1 day.**

**Phase 1 — Land Option B1 (cooperative yield).** Minimal, localized change to
the media-wait so the SD task stops hard-spinning while a DMA transfer is
in-flight. Preserve `TransferTask` FSM advancement; add a bounded yield. Target
outcome: eliminate the 5 kHz 12–13-byte tail-loss (#224 symptom). Re-soak
against the Session-24 SD ceilings to prove no regression. Ship it if it closes
#224. **~2–3 days incl. bench soak.**

**Phase 2 — Option C (double-buffer) *if* Phase 0 showed meaningful idle spin
AND Phase 1 left throughput on the table.** Ping-pong writeBuffer from the
coherent pool; overlap extraction with in-flight DMA. Gate on a measured
throughput target, not on principle. **~3–5 days.**

**Phase 3 — Option B2 / partial-A (true async submit) — DEFER.** Only if a
concrete product requirement (a specific SD streaming rate) is on the table that
Phases 1–2 provably cannot meet. Requires a non-blocking write-submit boundary
(a small, well-documented diskio patch exposing submit/poll to the manager) and
a WRITE_IN_FLIGHT sub-state with rigorous rotation/unmount quiescing. **Full
Option A (own the FAT cluster walk) is not recommended at any phase** absent a
requirement that justifies re-implementing filesystem write correctness.

**Recommendation in one line:** do **Phase 0 → Phase 1** now (closes the actual
ticket at low risk), hold Phases 2–3 behind a measured throughput requirement.
