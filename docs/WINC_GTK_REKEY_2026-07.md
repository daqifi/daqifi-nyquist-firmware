# WINC1500 AP-initiated GTK rekey starvation under sustained TX — investigation

Ticket: [#398](https://github.com/daqifi/daqifi-nyquist-firmware/issues/398)
Date: 2026-07 · Scope: research / analysis only, no code change, no bench.
Driver/FW version pinned in tree: **WINC1500 19.7.7**
(`config/default/driver/winc/include/drv/driver/m2m_types.h:115-123`).

Evidence tags per repo debugging discipline (CLAUDE.md → "Debugging
discipline"): **V** = verified from primary source (repo code / datasheet),
**E** = empirical test we ran, **I** = inference/hypothesis, **X** = external
authority, **N** = our own prior notes. This doc contains **no E** — no bench
was touched for it; the empirical anchors it cites are prior-session records
(**N**), not fresh measurements.

---

## 1. Question

Under sustained high TX (WiFi streaming / iperf2 over WPA2-PSK STA), can the
access point's **periodic group-key (GTK) rekey** handshake be starved or
dropped by our outbound traffic, causing the AP to de-authenticate the STA
(observed as mid-test `M2M_AUTH_FAILED` / throughput collapse)?

The ticket's working hypothesis: *"AP-initiated GTK rekey EAPOL-Key M1 frame
is queued behind our outbound `m2m_send` traffic in the WINC HIF, AP doesn't
see M2 ACK, de-auths the STA."*

**Verdict (summary): the hypothesis is directionally right that saturation is
the trigger, but wrong about the mechanism location.** The rekey handshake does
**not** traverse the host HIF at all — it is handled entirely by the closed
supplicant on the WINC. Our firmware's only exposure is *indirect*: sustained
host TX can saturate the WINC's internal shared packet memory and on-air TX
queue, and *that* is where the supplicant's EAPOL-Key M2 response can be
starved. Our host-side mitigations (#396 governor cap, #397 lifecycle
hardening) attack exactly this by keeping us below the saturation knee. See §6.

---

## 2. Where the rekey handshake actually runs (V)

**The WPA2 supplicant, including the 4-way handshake and the group-key (GTK)
2-way rekey handshake, runs on the WINC1500 itself — not on the PIC32 host.**

Evidence (V, by absence): a full grep of the host driver tree
(`config/default/driver/winc/**`) for any EAPOL / supplicant / 4-way /
group-key host API returns **nothing** in the 802.11 path:

- The only `handshake` symbols are TLS/ECC (`wdrv_winc_ssl.h`
  `WDRV_WINC_ECC_HANDSHAKE_INFO`) and `WPS` control — neither is the WPA2
  data-link key handshake.
- There is no `m2m_wifi_*` entry point that delivers or accepts an EAPOL-Key
  frame. The host API surface (`m2m_wifi.h`) exposes only connect / disconnect
  / scan / RSSI / socket data — the host never sees an EAPOL frame.

Corroboration (X): the Microchip *ATWINC15x0 Wi-Fi Network Controller Software
Design Guide* and the WINC1500 datasheet describe the WINC as an integrated
"network controller" that terminates the WPA/WPA2 supplicant on-chip; the host
only issues `m2m_wifi_connect(...)` with credentials and receives connection
state callbacks. (Cited from general Microchip WINC documentation; not a file
in this repo — treat as X, not V.)

**Consequence:** M1 (AP→STA) is consumed by the WINC firmware; M2 (STA→AP) is
generated by the WINC firmware. Neither crosses the SPI/HIF boundary to the
host. The ticket's phrasing "M1 queued behind our outbound `m2m_send`" cannot
be literally true — M1 is *inbound*, and the host is not in the EAPOL path.

## 3. The host HIF path, and why it is serialized (V)

Even though rekey is on-chip, it is worth pinning the host-side transport
mechanics, because they bound how fast we can flood the chip and how promptly
the host services chip events.

**Single binary semaphore serializes host TX and host RX servicing** (V):

- `m2m_hif.c:55` — one `hifSemaphore` (binary, count 1).
- `hif_send()` (`m2m_hif.c:266-386`) pends `hifSemaphore` **for the entire
  transaction**: `hif_chip_wake()` → allocate WINC RX DMA slot (busy-polls
  `WIFI_HOST_RCV_CTRL_2` up to 1000×, with a `nm_sleep(1)` slowdown after 500
  iterations) → `nm_write_block` the payload → `hif_chip_sleep()` → post.
- `hif_isr()` (`m2m_hif.c:404-537`), which drains inbound frames (data, and the
  connection-state-changed events), pends the **same** `hifSemaphore`.

So a host `send()` and the host's servicing of an inbound frame are mutually
exclusive at the SPI layer. This is by Microchip design (single SPI master to
the chip), not a bug.

**Host TX entry path in our firmware** (V):

- `socket.c:send()` → `SOCKET_REQUEST(SOCKET_CMD_SEND|M2M_REQ_DATA_PKT, ...)`
  → `hif_send(...)`. When the chip cannot allocate an internal packet buffer,
  `hif_send` returns `M2M_ERR_MEM_ALLOC`; `send()` maps that to
  `SOCK_ERR_BUFFER_FULL` (`socket.c:824-855`).
- Our `wifi_tcp_server.c` is **non-blocking**: on `SOCK_ERR_BUFFER_FULL` it
  leaves data in `writeBuffer` and retries on the next
  `TransmitBufferedData()` (`wifi_tcp_server.c:117-146`); the completion
  `SOCKET_MSG_SEND` callback re-arms the next send
  (`wifi_manager.c:730-780`).

**Task priorities that matter** (V, from CLAUDE.md Task Priority Map):

- `streaming_Task` (pri **6**) issues the `send()` calls during a WiFi stream.
- `app_WifiTask` (pri **2**) runs the TCP-SCPI dispatch and also sends.
- `lWDRV_WINC_Tasks` (pri **1**) calls `m2m_wifi_handle_events()` →
  `hif_handle_isr()` (`m2m_wifi.c:502`) — i.e. the host's servicing of inbound
  chip frames and connection-state events runs at the **lowest** app priority.

**I (inference):** during a saturating stream, the pri-6 encoder/send loop both
(a) monopolizes CPU relative to the pri-1 WINC task and (b) repeatedly holds
`hifSemaphore`, so the host services inbound chip events (including the
CONN_STATE_CHANGED that reports a disconnect) with latency. This delays our
*observation and handling* of a rekey-induced disconnect; it does **not**
delay the on-chip M2, so it is not the *cause* of a rekey failure. It is,
however, why a disconnect can look "sudden" from the host's perspective.

## 4. The real starvation locus: WINC internal buffers / on-air TX queue (I)

Because the supplicant is on-chip (§2), the contention that could starve a GTK
rekey is **inside the WINC**, between:

1. Host data frames the WINC has accepted (they occupy the WINC's shared
   internal packet memory and sit in its 802.11 TX queue awaiting airtime),
   and
2. The supplicant's EAPOL-Key **M2** response, which must also allocate an
   internal buffer and win airtime to be transmitted before the AP's M1
   retransmit budget expires.

**Mechanism (I, grounded in the V facts above):** at/above the saturation knee
the host keeps `hif_send` returning success right up until the WINC's internal
memory is full (then `SOCK_ERR_BUFFER_FULL`). A WINC whose TX queue is
backlogged with host data has minimal headroom/airtime for a time-critical
EAPOL-Key frame. If M2 is delayed past the AP's M1 retry window, the AP
de-auths → the host sees a CONN_STATE_CHANGED DISCONNECTED with
`WDRV_WINC_CONN_ERROR_AUTH`, decoded as **"Auth Failed"**
(`wifi_manager.c:509-514`). That is exactly the `M2M_AUTH_FAILED` symptom the
ticket and #385 describe.

**This remains I, not V/E.** Confirming it requires the instrumentation the
ticket lists and which this doc did **not** perform:
- Wireshark promiscuous capture on the AP channel showing `M1 → (no M2) →
  de-auth` time-correlated with a saturating stream (E we do not yet have).
- WINC internal TX-queue occupancy at the moment of M1 — not directly
  observable from the host without vendor instrumentation (the FW is a closed
  blob, `m2m_aio_3a0.fw`).

The closest host-observable proxy we already have is the **partial-send
counter** — see §5.

## 5. Symptoms to watch for, and existing host-observable proxies

### 5.1 Direct symptom
- Mid-stream disconnect logged as
  `WiFi STA Disconnected - Error Code: <n>, ... Reason: Auth Failed`
  (`wifi_manager.c:528`), `errorCode == WDRV_WINC_CONN_ERROR_AUTH`.
- Correlates in prior sessions (N) with the #385 "~30 disconnects/test under
  heavy load" and the general WiFi-variance history.

### 5.2 Partial-sends as a leading indicator (N + V)
`SYST:STR:STATS?`-adjacent WiFi diagnostics expose
`WifiTcpPartialSends` / `WifiPartialBytesMissing`
(`wifi_tcp_server.h:86-88`; incremented `wifi_manager.c:756-759`; surfaced in
`SCPIInterface.c:3019-3021`). A partial send = `SOCKET_MSG_SEND` reported fewer
`sentBytes` than requested — i.e. the WINC accepted less than a full segment,
the softest observable sign of WINC send-side back-pressure.

Prior-session data (N, ticket **#500**, still OPEN): even on *CLEAN* 1-hour
WiFi soaks, `wifi_partial_sends` runs a steady ~480-595/hr (≈0.13-0.17/s) with
zero drops; at 2 kHz 5×T1 it jumped to ~1,661/hr and *did* begin to correlate
with `wst_steady` byte loss. **I:** partial-sends are plausibly the same
internal-buffer-pressure signal that, taken far enough, also starves a rekey
M2. #500's "find the knee" plot is the natural place to look for a
rekey-disconnect threshold too — they may share a root cause (WINC internal TX
pressure). This is a hypothesis linking two N datasets, not a proven identity.

### 5.3 Silent re-association (V + N)
Our code already anticipates a WINC that "silently re-associates (AP rekey,
brief deauth handled internally) without firing a CONN_STATE_CHANGED"
(`wifi_manager.c:192-196`) and repairs the drifted `STA_CONNECTED` flag via a
5-s RSSI reconcile probe. **I:** a *successful* fast GTK rekey may cause exactly
this benign flag drift (device looks "Disconnected" for a few seconds while
pings/iperf keep working); a *failed* rekey instead produces the hard
`Auth Failed` disconnect in §5.1. Distinguishing the two on the bench is a
useful diagnostic: reconcile-repaired vs hard-`AUTH` disconnect.

### 5.4 Variance-campaign correlation (N)
The WiFi variance campaign (memory `project_wifi_variance_campaign`, 2026-06-14)
found that the cells that swing most are low-channel / high-rate — the ones
closest to the wire ceiling (i.e. those most likely to saturate WINC TX). If
rekey starvation is real, disconnects should concentrate in exactly those
high-rate cells and be rare in the multi-channel cells that sit below the link.
That is a falsifiable prediction to check against future soak logs.

## 6. Are we exposed? (I, qualified)

**Partially, and only above the saturation knee — which our shipped caps are
designed to stay below.**

- The gating pre-requisites the ticket set for a Microchip escalation are now
  **met**: **#396** (characterize stable ceiling + auto-backoff) is **CLOSED**,
  **#397** (harden mid-stream disconnect — never present a wedged device) is
  **CLOSED**. So the "rule out our-side causes first" bar is cleared on paper.
- The enforced WiFi caps are **worst-night-observed by policy** (CLAUDE.md →
  Streaming Frequency Capping; memory `project_wifi_variance_campaign`). Their
  purpose is precisely to hold TX below the rate at which WINC internal
  buffers saturate. **I:** so at the *enforced* caps we expect to be below the
  rekey-starvation regime; the historical `M2M_AUTH_FAILED` bursts (#385) were
  seen under *uncapped* / at-ceiling load, consistent with §4.
- We have **no patching ability** over the WINC FW blob (`m2m_aio_3a0.fw`); if
  §4 is the true mechanism, a definitive fix is Microchip's, not ours. This is
  the ticket's own "filed defensively" conclusion.

**What is NOT yet done** (so exposure is not fully characterized):
- No Wireshark `M1→no-M2→deauth` capture time-aligned to a saturating stream
  (the decisive E).
- No standalone Harmony WINC1500 iperf demo repro (#377) proving it is not our
  code. Until that exists, an escalation would be premature per the ticket's
  own checklist item 3.

## 7. Mitigations (all follow-up PRs — no code in this doc)

Ordered by leverage and by how much is in our control.

1. **Stay below the saturation knee (already shipped, primary).** The #396
   governor cap is the real mitigation: keep host TX under the rate that
   backlogs WINC internal TX. Action item: *quantify* the knee vs
   partial-sends (extend #500's plot) and confirm the enforced WiFi caps sit
   below the rekey-disconnect rate, not just below the byte-loss rate. Firmware
   change: none if caps already conservative; possibly a small derate on the
   high-rate low-channel cells the variance campaign already flagged as
   cap-optimistic.

2. **TX pacing during the rekey window (host, speculative — needs the E
   first).** If a capture shows disconnects cluster right at rekey boundaries,
   a brief host-side send pause could give the WINC airtime for M2. BUT the
   host cannot see M1 (§2), so it cannot know when a rekey is happening — this
   is only feasible if a rekey-interval-aligned periodic pacing proves
   effective, which is fragile. Lower priority; do not build before §6's E
   exists.

3. **AP-side GTK rekey interval (config, not always available).** Lengthening
   the AP's group rekey interval widens the window in which M2 must land. This
   is a security/robustness trade-off and is outside firmware — a
   documentation/deployment note for customer APs, and the standard first
   question in a Microchip forum thread. Not a firmware PR.

4. **Driver-level handling (lowest leverage).** The WINC RX/event servicing
   runs at pri 1 (§3); we already avoid adding fast-path delay to
   `lWDRV_WINC_Tasks` (memory `feedback_winc_task_timing` — a 1 ms delay there
   bricked USB CDC in #335). There is no safe host-driver lever that
   accelerates an on-chip supplicant. Any change here is high-risk / low-yield.

5. **Escalation to Microchip (record-keeping, per ticket's defensive framing).**
   With #396/#397 closed, the remaining gate is the standalone-demo repro
   (#377). If/when that exists plus a pcap, open a support case + forum thread
   with: WINC FW 19.7.7, HIF queue/partial-send measurements, the pcap, and the
   demo repro. Realistic outcome per the ticket: we live with the cap-based
   workaround and keep #398 as the public record.

---

## 8. Bottom line

- **V:** GTK rekey is handled entirely on the WINC; the host is not in the
  EAPOL path (no host EAPOL API exists in the tree). The host HIF serializes
  TX and RX-servicing through one binary semaphore, and RX-servicing runs at
  the lowest app priority.
- **I:** the plausible starvation locus is *inside* the WINC — host data frames
  backlogging its internal TX queue can delay the supplicant's M2 past the AP's
  M1 retry budget → `AUTH` de-auth. Exposure is confined to the
  above-saturation regime; the shipped worst-night caps are designed to stay
  below it.
- **N:** symptom to track = `Auth Failed` disconnects (`wifi_manager.c:528`);
  leading indicator = `WifiTcpPartialSends` (#500), which may share a root
  cause; concentration expected in the high-rate low-channel cells the variance
  campaign flagged.
- **Not done (needed before escalation):** the decisive Wireshark
  `M1→no-M2→deauth` capture and the standalone Harmony demo repro (#377). Until
  those exist this remains an I-level mechanism, and a Microchip escalation is
  premature per the ticket's own gating checklist.

*No firmware changes are proposed by this document. Mitigations §7.1-§7.5 are
tracked as follow-up work.*
