# Self-Describing SD Logs with Metadata Header (Issue #188)

## Context

SD `.bin` (protobuf) files do not include `TimestampFreq` or device identification metadata. Downstream parsers cannot reconstruct timestamps or identify the device without a live connection. The issue requires every SD log file to be fully parseable offline.

**Current state by format:**

| Format | Metadata header? | TimestampFreq? | Split file headers? |
|--------|-----------------|----------------|---------------------|
| CSV | Yes (device, serial, tick rate) | Yes (`TimerApi_FrequencyGet`) | No |
| JSON | Yes (device, serial, tick_hz) | Yes (`TimerApi_FrequencyGet`) | No |
| Protobuf | **No** | **Bug**: uses `TimerApi_CounterGet` (counter value, not frequency) | N/A |

## Bug Fix

`NanoPB_Encoder.c:529` — `timestamp_freq` field populated with `TimerApi_CounterGet()` (returns current timer counter value) instead of `TimerApi_FrequencyGet()` (returns clock frequency in Hz). CSV and JSON encoders use the correct function.

## Thread Safety

The `gSdPbMetadataSent` bool follows the same pattern as existing `csvHeaderSent` and `jsonHeaderSent`:
- Set to `true` in streaming task (priority 2)
- Set to `false` from SCPI commands (priority 7) only when streaming is stopped/restarting
- `bool` is atomic on PIC32MZ (32-bit aligned word access)
- Worst case: metadata written twice to a split file (harmless — protobuf decoders take last value)
